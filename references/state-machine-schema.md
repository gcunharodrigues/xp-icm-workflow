# State Machine Schema (L1 — `<workspace>/CONTEXT.md`)

> **Path resolution:** paths `scripts/` in this document refer to `<SKILL_DIR>/scripts/`, where `SKILL_DIR` is defined in L0 (`CLAUDE.md`).

> Canonical schema of the workspace root `CONTEXT.md`. Validated on every session's pre-flight check. Property-based tested in `tests/unit/test_state_machine.py`.

## File format

`<workspace>/CONTEXT.md` is Markdown with **YAML frontmatter** followed by narrative sections. The frontmatter is the source of truth for state; the body is for human readability only.

```markdown
---
<frontmatter yaml>
---

# Workspace NNN — <slug>

(optional body, narrative)
```

## Frontmatter — mandatory fields

| Field | Type | Values / format | Origin |
|---|---|---|---|
| `workspace` | string | `NNN-slug` (e.g.: `042-feat-auth`) | Bootstrap |
| `profile_base` | string | one of 11 canonical profiles | Bootstrap |
| `profile_effective_hash` | string | sha256 hex (64 chars) | `<SKILL_DIR>/scripts/profile-merge.py` |
| `tier` | string | `experimental` \| `tool` \| `development` \| `production` | Bootstrap |
| `project_root` | string | absolute path | Bootstrap (CWD or `--project-root`) |
| `base_branch` | string | valid git branch name | Bootstrap (`git rev-parse --abbrev-ref HEAD`) |
| `workspace_branch` | string | `workspace/NNN-slug` | Bootstrap |
| `stage_atual` | string | `00` \| `01` \| ... \| `08` | Current session |
| `sub_stage` | string | enum (see §Sub-stage enum) | Current session |
| `status` | string | enum (see §Canonical statuses) | Current session |
| `iteration` | integer | ≥ 0, monotonic | `0` at bootstrap; `++` at stage 08 exit B |
| `history` | list | append-only (see §History) | Every transition appends |
| `last_action` | string | free-form description | Current session |
| `last_action_at` | string | ISO 8601 timestamp | Current session |
| `next_action` | string | free-form description | Current session |
| `block_reason` | string | `human_gate` \| `stop_point` \| `error` \| `hitl` \| `lead_resolution` | Current session (mandatory if status=BLOCKED, absent otherwise) |
| `last_transition` | object | `{from, to, at, commit_sha}` | Every transition |

## Frontmatter — optional fields (nullable / omittable)

| Field | Type | When present | Default |
|---|---|---|---|
| `stages_skipped` | list of strings | always (bootstrap determines via profile/tier) | `[]` |
| `logs_root` | string \| null | system with external logs (H3) | `null` |
| `waves` | object \| null | `stage_atual >= 04` (R2.8) | absent before 04 |
| `llm_review_skipped_count` | integer | whenever ≥1 skip occurs | `0` |
| `spawn_from` | string | workspace was generated by stage 08 exit C | omitted |
| `spawn_to` | string | workspace generated another via stage 08 exit C | omitted |
| `custom_stop_points` | list | yaml override declared (D3) | omitted |
| `prev_outputs` | list of strings | v4.0 handoff — paths of outputs delivered by previous session | `[]` |
| `pending` | list of strings | v4.0 handoff — next tasks/concerns for current session | `[]` |
| `revisit_after` | string | yaml override declared (Q16) | omitted |

## Canonical statuses (3 values — v4.0)

| Status | When | Typical next action |
|---|---|---|
| `IN_PROGRESS` | active session working (includes lead resolution, subagent dispatch) | continue work or transition |
| `BLOCKED` | any blocking condition — human gate pending, stop point triggered, error, HITL awaiting | human resolves; `IN_PROGRESS` |
| `COMPLETED` | entire workspace closed (stage 08 exit A) | none — workspace archived |

### `block_reason` field (mandatory when status=BLOCKED, absent otherwise)

| Reason | Former status (v3) | Meaning |
|---|---|---|
| `human_gate` | COMPLETED_AWAITING_HUMAN | Stage done, waiting for human approval |
| `stop_point` | BLOCKED_STOP_POINT | A/B/C menu triggered |
| `error` | BLOCKED_ERROR | Runtime/CI/merge failure |
| `hitl` | BLOCKED_HITL | Mixed wave, HITL task awaiting human |
| `lead_resolution` | LEAD_RESOLUTION_IN_PROGRESS | Lead executing RETRY/VOID for stuck task |

**Special variation:** `RESTARTING_AT_PHASE_X` (H1) is recorded in `history` as an `iteration++` event, but the `status` itself reverts to `IN_PROGRESS` at stage X with `iteration` incremented.

## Sub-stage enum (R2.6 — v4.0 simplified)

| Stage | Valid values |
|---|---|
| 00 Recon | `00_in_progress`, `00_completed` |
| 01 Discovery | `01_in_progress`, `01_completed` |
| 02 Design+Plan | `02_in_progress`, `02_completed` |
| 04 Implementation Waves | `04_wave_<N>_in_progress`, `04_wave_<N>_completed` (N positive integer) |
| 08 Feedback Intake | `08_in_progress`, `08_decided_A`, `08_decided_B`, `08_decided_C` |

**Rule:** `sub_stage` ALWAYS starts with prefix `<stage_atual>_`. Mismatch = inconsistency → Recovery Wizard.

## `history` — append-only

List of events. Chronological order preserved. Each item:

```yaml
- at: "2026-04-25T14:30:00Z"
  event: "stage_transition"  # see §Event types below
  from: "02_completed"        # optional, depends on event
  to: "03_in_progress"        # optional, depends on event
  commit_sha: "abc123def"     # optional, if there was a commit
  note: "stack approved by human"  # optional free-form text
```

**Rule:** sessions NEVER edit existing items; only append. Recovery Wizard may prepend a `recovery_applied` item documenting reconstruction.

### Canonical event types

| Event | Description | Required fields |
|---|---|---|
| `stage_transition` | Transition between stages/sub_stages | `from`, `to`, `commit_sha` |
| `stop_point_triggered` | Stop point triggered | `stop_point_id`, `from` |
| `stop_point_resolved` | Human answered stop point | `stop_point_id`, `to` |
| `iteration_increment` | Stage 08 exit B: `iteration++` | `from`, `to`, `commit_sha` |
| `recovery_applied` | Recovery Wizard corrected inconsistency | `inconsistency_type`, `action` |
| `wave_completed` | Wave N of stage 04 finalized | `wave`, `commit_sha` |
| `blocked_error` | Runtime error blocked progress | `error_type`, `message` |
| `workspace_bootstrapped` | Initial workspace bootstrap | `commit_sha` |
| `feedback_session_started` | Human opened feedback session (stage 08) | `from` |

### `error_type` values (known, growing list — not an enum)

When the `blocked_error` event is appended to `history` (and `status: BLOCKED_ERROR`
is set in parallel), the `error_type` (free-form string) field of the event is
populated with one of the values below. List evolves per version; no automatic
enforcement. Schema of `last_transition` remains `{from, to, at, commit_sha}`
without `error_type` — this lives only in the event row of `history`.

- `merge_conflict`
- `ci_red`
- `cap_exceeded`
- `cleanup_unsafe`
- `runtime_cleanup_failed`
- `forensic_max_retries`        <!-- v3.8.0 — cap MAX_FORENSIC_RETRIES exhausted -->
- `forensic_script_crash`       <!-- v3.8.0 — forensic-plus.py exit 1 -->
- `lead_resolution_audit_failed`        <!-- v3.9.0 — stage 05 audit detected issues -->
- `lead_resolution_failed`              <!-- v4.0 — RETRY+VOID both exhausted -->
- `lead_decision_missing`               <!-- v3.9.0 — lead chose bucket without registering lead-decision.md -->
- `critic_unavailable`                  <!-- v3.9.0 — Agent tool fail / quota exhausted after retry -->
- `critic_abstain_loop`                 <!-- v3.9.0 — 2 consecutive critic ABSTAIN -->
- `e2e_suite_failed`                    <!-- v3.10.0 — L4 wave gate e2e step 11b red -->
- `e2e_suite_missing`                   <!-- v3.10.0 — Stage 05 audit detected suite absent in profile that requires it -->
- `e2e_suite_stale`                     <!-- v3.10.0 — Stage 05 audit suite > 7 days with user-facing tasks delivered -->
- `e2e_skip_unjustified`                <!-- v3.10.0 — Stage 05 audit `**E2E:** skip` without rationale -->
- `human_abort`

## `waves` — schema (present only if `stage_atual >= 04`)

```yaml
waves:
  current: 2                     # int, wave in progress
  completed: [1]                 # list of int, waves already merged
  current_sub_wave: null         # optional str, e.g. "2.a" if subdivided (E3)
  blocked_at_sub_wave: null      # optional str, set if BLOCKED_ERROR mid-wave (E4)
  blocked_task: null             # optional str, slug of blocked task
```

## `last_transition` — schema

```yaml
last_transition:
  from: "02_completed"
  to: "03_in_progress"
  at: "2026-04-25T14:30:00Z"
  commit_sha: "abc123def456"
```

**Rule:** `commit_sha` must exist in `git log` of `workspace_branch`. If it does not exist (force push or branch reset) → Recovery Wizard inconsistency #4 (R2.7).

## Inconsistency heuristics (Recovery Wizard — R2.7)

Pre-flight of a NEW session triggers Recovery if any of the following:

1. `profile_effective_hash` does not match hash recomputed from `<workspace>/_config/profile-effective.yaml`
2. Outputs declared in `history` do not exist on the FS
3. `status: IN_PROGRESS` without commit in `workspaces/NNN/*` in the last 24h
4. `last_transition.commit_sha` does not exist in git history
5. `waves.current` declares wave N but branches `wave-NNN-N/<task>` are absent

Each inconsistency has a specific message + proposed action in `references/recovery-wizard.md`.

## Validation

`<SKILL_DIR>/scripts/validate-state.sh` called on every session's pre-flight:

1. Parse YAML frontmatter (PyYAML strict load).
2. Validate mandatory fields present.
3. Validate enums (`tier`, `status`, `sub_stage` — prefix matches `stage_atual`).
4. Validate `iteration >= 0` and monotonic vs `history`.
5. Validate `waves` absent/null if `stage_atual < "04"`; present if `>=`.
6. Validate `last_transition.commit_sha` exists in git history.
7. Recompute `profile_effective_hash` and compare.
8. Run Recovery heuristics (R2.7).

Failure in any step → specific message + abort OR Recovery Wizard.

## Property-based testing (`tests/unit/test_state_machine.py`)

Hypothesis generates random states respecting schema. Property invariants:

- Every valid transition preserves YAML schema.
- `iteration` is monotonic (never decreases).
- `history` is append-only (never shrinks).
- `sub_stage` always prefixed by `stage_atual`.
- `waves` absent iff `stage_atual < "04"`.
- Status ∈ canonical enum.
- `last_transition.from` is the previous `sub_stage`, `last_transition.to` is the current `sub_stage`.
