# Session Handoff Protocol

> **Version:** v4.0 simplified. Previous v3.x protocol (Phase 1/2 gate, KICKOFF blocks, dashboard updates) preserved below as legacy reference.

## v4.0 Simplified Handoff

**1-stage-1-session + 1-wave-1-session** remains. Key changes from v3:

1. **L1 is the handoff mechanism.** `prev_outputs` (list of output paths) and `pending` (list of next tasks/concerns) in L1 frontmatter carry session state forward. `_kickoff.md` is being phased out — new v4 workspaces use L1 fields; v3 workspaces may still have legacy kickoff files.
2. **Statuses simplified.** `BLOCKED` with `block_reason` replaces `COMPLETED_AWAITING_HUMAN`, `BLOCKED_STOP_POINT`, `BLOCKED_ERROR`, `BLOCKED_HITL`. `LEAD_RESOLUTION_IN_PROGRESS` absorbed into `IN_PROGRESS`.
3. **Human gate uses `BLOCKED` + `human_gate`.** Work done → set `BLOCKED`/`human_gate` → commit → print gate → wait for human → set `IN_PROGRESS` → commit → EXIT.
4. **Stage 04 last-wave gate → 08.** No separate stages 05/06/07. CI/E2E/merge happen as inline gates within stage 04. Last wave transitions directly to stage 08 (feedback).
5. **Project-root CLAUDE.md** maintained by `handoff.py update-project-md` as external dashboard.

Stages 00/01/02 may still generate `_kickoff.md` for backward compatibility with v3 workspaces. The canonical source of session state is L1 `prev_outputs`/`pending`.

## Legacy v3.x Protocol (preserved for reference)

> Canonical: **1 stage = 1 session**. Each stage ends with a dual handoff (verbal + persisted file) and the current session EXITS. The next session starts fresh — lean context, cache miss accepted as cost.

## Why 1-stage-1-session (supersedes Q3 batched)

Empirical observation from real usage (beta1/beta2): batched sessions (design 00+01+02 in one, closing 05+06+07 in one) grew context beyond the 2-8k token target per L2. Total token spend grew nonlinearly vs number of sub_stages.

Trade-off resolved in favor of fresh cache:

| Option | Token cost | Cache miss | Context pollution |
|---|---|---|---|
| Batched (Q3 v1) | lower (1 cache miss per batch) | 1 per batch | high — sub_stages accumulate |
| **1-stage-1-session (v2)** | 1 cache miss per stage = 8 misses in the full flow, but each session starts ~2-3k tokens | 8 total | minimal — fresh session per stage |

User signaled that the gain in fresh context outweighs the cache miss cost. Decision revised: **A1 — ALL stages 00→07 uniform = 1 session each**.

Stage 04 keeps its wave-aware version: 1 lead session per wave (status quo, decision 2a). Sub-waves do not trigger an explicit handoff — lead persists across sub-waves within the same session.

### Why inline gate (bug v3.4.2)

Before v3.4.2, end-of-stage executed commit + render kickoff + print exit + EXIT without waiting for the human gate. Symptom: new session opened, detected `status=COMPLETED_AWAITING_HUMAN`, asked for approval, and re-printed the kickoff — confusing loop. Cause: contradiction between "Gates" section (human approves BEFORE transition) and "End of stage handoff" (transition WITHOUT waiting). Fix: split into two phases within the same session; kickoff only renders after approval.

### Response "adjust X" / "abort"

- **"adjust X"**: L1 status reverts to `IN_PROGRESS`, history appends `gate_rejected`, sub_stage stays `<NN>_completed` (reverts to `<NN>_in_progress` if change is non-trivial). Session returns to work with the request. When outputs are redone, Phase 1 repeats.
- **"abort"**: L1 status `BLOCKED_ERROR`, history `human_abort`. Commit + exit. Workspace stays in `BLOCKED_ERROR` awaiting manual intervention.

## Verbal handoff (session-end)

From v3.1.0 onwards, the verbal handoff is simplified — the ICM region in
`<project_root>/CLAUDE.md` (maintained by `handoff.update_project_claude_md`)
eliminates the need for the KICKOFF copy-paste block. The next fresh session opened
in the project_root loads CLAUDE.md automatically and has the full read order +
workspace state.

End of session prints:

```
✅ Stage 02 (design) COMPLETE — workspace 042-feat-auth (commit abc123de)
   - L1: stage_atual=03, sub_stage=03_in_progress, status=IN_PROGRESS
   - Outputs: stages/02_design/output/plan.md (8 tasks)
              stages/02_design/output/decisions.md (2 ADRs)
   - Kickoff: stages/03_wave_planner/_kickoff.md generated
   - CLAUDE.md updated: <project_root>/CLAUDE.md (ICM region)

🔄 To continue: close this session (Ctrl+D or /exit) and open a new
   Claude session at the project_root. CLAUDE.md handles the rest.
```

Distinct layers, not redundant:
- `<project_root>/CLAUDE.md` (ICM region): external state dashboard, read
  automatically by Claude Code in a fresh session. Canonical doc:
  `references/project-root-claude-md.md`.
- `_kickoff.md` in the next stage: transitional detail (prev_outputs,
  prev_decisions_summary, pending_for_this_stage). Persistent, read after L0/L1/L2.

## Stage 08 exits A/B/C and CLAUDE.md root

| Exit | Function in `handoff.py` | State after exit |
|---|---|---|
| **A** (close) | `remove_workspace_block(workspace, ...)` | Workspace block removed. If it was the last active: ICM region replaced by "none active + run /init" message. |
| **B** (restart stage X, iteration++) | `update_project_claude_md(workspace, stage_target=X, iteration=N+1, ...)` | Block updated showing new stage and iteration. |
| **C** (spawn new workspace) | Session A: `remove_workspace_block(workspace_owner, ...)`. Session B (bootstrap): `update_project_claude_md(new_workspace, ...)` adds block for 043. | Owner block removed. Bootstrap in separate session adds new block. |

## Atomicity and concurrency (G15, G12)

All writes to `<project_root>/CLAUDE.md` use the write-tmp + fsync +
rename pattern (atomic on POSIX and Windows NTFS). A crash mid-write does not
corrupt the original file.

Concurrency: workspace branch isolates sessions via git. A write conflict
generates a git error → session aborts with `BLOCKED_ERROR`. Recovery wizard
(`CLAUDE_MD_ROOT_STALE`) detects and regenerates the block from L1.

## Contract with future `/init` (G4)

`/init` regenerates CLAUDE.md from code. Markers
`<!-- ICM-START -->`/`<!-- ICM-END -->` delimit the skill's exclusive region.
**Do not invoke `/init` while an ICM workspace is active** — explicit warning in
the region itself. After Exit A of the last workspace, `/init` operates freely
(the ICM region has already been replaced by the idle message).

### Exit A migrates CLAUDE.md root to base branch (v3.4.1)

From v3.4.1 onwards, when Exit A (close) finalizes the last active workspace,
`deactivate_project_claude_md` performs two writes:

1. `<project_root>/CLAUDE.md` (workspace branch tree) — idle region.
2. `<project_root>/.icm-main/CLAUDE.md` (base branch via worktree) — same
   content + commit in base.

Without step 2, the root CLAUDE.md would disappear when the workspace branch is
deleted after archiving, leaving the project_root without any persistent ICM
dashboard. Canonical doc: `references/project-root-claude-md.md`
(section "Owner transition at exit A").

## Stage 04 (waves) — exception

Stage 04 = **1 lead session per wave**. Each wave ends with a handoff to the next wave session. Two cases:

### Case A — Mid-wave handoff (wave N → wave N+1, automatic, no human gate)

Applies when more waves remain after the current wave completes.

1. All subagents in wave N delivered COMPLETE.
2. Wave-reviewer approved (L2 forensic+ + L3 critic).
3. Sequential merge into `{{BASE_BRANCH}}` + global CI green.
4. Write `output/wave-<N>/wave-summary.md`.
5. Update L1: `sub_stage = 04_wave_N_completed → 04_wave_N+1_in_progress`. Status stays `IN_PROGRESS`.
6. Render `_kickoff.md` in the same stage 04 directory (`stages/04_implementation_waves/_kickoff.md`) with updated `prev_outputs`, `pending` (next wave tasks), `prev_state_prose`.
7. Atomic commit: `workspace <NNN>: wave <N> completed, kickoff for wave <N+1>`.
8. **EXIT the session.** Wave N+1 starts in a FRESH session with clean context and new pre-flight.

**Anti-pattern:** running wave N+1 in the same session as wave N. Context accumulates, errors compound. The handoff forces a checkpoint (CI verified, summary written, L1 updated) before the next wave begins.

### Case B — Last-wave handoff (wave N → stage 05, WITH human gate)

Applies after the LAST wave completes (no more waves in wave-plan.md).

1. Same steps 1-4 as Case A (subagents complete, reviewer approves, merge, CI green, wave-summary).
2. Update L1: `sub_stage = 04_wave_N_completed`, `status = COMPLETED_AWAITING_HUMAN`.
3. **Gate inline.** Lead pauses and presents to human:
   - Wave-summary.md summary
   - Pending items (if any)
   - Next stage (08_feedback_intake)
4. Human approves → Phase 2 GATE_APPROVED:
   - Update L1: `stage_atual = 08`, `sub_stage = 08_in_progress`, `status = IN_PROGRESS`. (v4.0: skips 05/06/07 — goes directly to feedback intake)
   - Atomic commit.
   - Print KICKOFF block.
5. EXIT the session. Stage 05 starts in a fresh session.

Cross-ref: stage 04 L2 template `## End of stage handoff` section has the wave-aware details (Cases A/B with exact commands).

## Stage 08 (feedback intake) — real terminal

The workspace only closes (`status: COMPLETED`) in one of the 3 exits of stage 08, decided by **inferring the intent from the human's free-form feedback** (no explicit A/B/C menu). Session 08 maps feedback heuristics → A/B/C, computes confidence, mini-confirms with the human before executing.

**Exits:**

- **A — Close:** workspace `COMPLETED`. Lessons recorded in `docs/lessons.md`. Session exits WITHOUT generating kickoff.
  - Inference signals: "all good", "close", "no feedback", silence.

- **B — Restart stage X:** L1 `iteration++`, `stage_atual=X`, `sub_stage=X_in_progress`. Canonical bug→stage mapping (tests→05, code→04, design→02, etc.). GENERATES kickoff for stage X. Prints KICKOFF verbally.
  - Inference signals: "bug in X", "broke Y", "regression", explicit mention of stage.

- **C — Spawn new workspace:** L1 `status=COMPLETED`, `spawn_to=<new-slug>`. Prints instruction for user to invoke `/xp-icm-workflow project-root=<X> spawn_from=<NNN>` in a new session. Does NOT generate kickoff (new workspace = different one).
  - Inference signals: "pivot", "new project", "big new feature", profile/tier change.

Detailed heuristics in the L2 of stage 08 (`stages/08_feedback_intake/CONTEXT.md` section "Intent inference"). Confidence < 0.6 → session asks a short clarification before inferring.

Mini-confirm template (1 single menu):

```
Understood: <inferred exit in prose, 1-2 sentences>

[y] confirm and execute
[n] cancel, ask again
[edit] describe the adjustment
```

L1 final status = `COMPLETED`. Workspace branch can be archived (see R3.4).

## Stage 08 trigger

User returns by opening a new session (paste prompt from the KICKOFF block of stage 07). No special command — the "1 stage = 1 session" protocol treats stage 08 like all others, except that **the input is free-form feedback** instead of following a deterministic L2 flow.

Time between 07→08: undefined. Workspace stays in `COMPLETED_AWAITING_HUMAN` while the user uses the project in real life.

## Anti-patterns

### Continuing a stage beyond its declared scope

The current session MUST NOT advance to the next sub_stage / stage beyond what is declared. Even if it seems "quick to do it all at once" — the point of the rule is lean tokens. If the urge to advance arises: stop, handoff, open a new session.

### Skipping handoff because "it's obvious"

A trivial stage (e.g., 01 short discovery) still MUST follow the protocol. Otherwise handoff becomes optional and users forget it in real stages. Uniformity > marginal efficiency.

### Verbal kickoff without file

Only printing verbally without generating `_kickoff.md` breaks traceability. The next session needs the file to:
- Understand what was delivered (`prev_outputs`)
- Handoff-marked pending items (`pending_for_this_stage`)
- Absolute read order (paths may change between sessions)

### Migrating beta1/beta2 workspaces

Decision **4B**: no forced migration. Workspaces created before beta3 remain in batched mode. Do not attempt to convert — risk of breaking L1↔outputs atomicity. Those workspaces finish in the original batched model.

## Handoff commit atomicity

At the end of the stage, the commit MUST include simultaneously:

1. Outputs of the current stage (`stages/<NN>/output/*`)
2. Updated L1 (`workspaces/<NNN>/CONTEXT.md`)
3. `_kickoff.md` for the next stage (`stages/<NN+1>/_kickoff.md`)

Pre-commit hook validates outputs↔L1 atomicity (already exists). Hook does **NOT** validate `_kickoff.md` automatically — convention depends on the session. But hook accepts kickoff because it is inside `workspaces/NNN/stages/`.

Commit message (regulated by the commit-msg hook):

```
workspace 042: stage 02 complete + kickoff stage 03
```

## Validation

Pre-flight check for the next session:

1. Reads L0 + L1.
2. Confirms `last_transition.to == "<NN+1>_in_progress"`.
3. Reads `stages/<NN+1>/_kickoff.md` if it exists; warning if absent (may have been v1 generation or bug — recovery wizard can prepend an audit entry).
4. Continues with the L2 of the current stage.

`scripts/handoff.py` (Wave 2/3 fix) implements render + validation. Tests in `tests/unit/test_handoff.py`.

## Mid-Session Context Checkpoint (anti-compact)

### Problem

The context window compacts governance files (L0, L1, L2, kickoff) read at the start of the session. The agent loses critical instructions — branches, TDD protocol, read order, stop points. Result: agent ignores ICM and operates outside the protocol.

### Rule: checkpoint after each completed task

In **stage 04 (implementation waves)**, after each task completes (TDD cycle step 7 COMPLETE), the lead MUST:

1. **Update `_kickoff.md`** with cumulative progress — add `prev_outputs` for the completed task, update `pending_for_this_stage`.
2. **Update L1** (`CONTEXT.md`) if a sub_stage transition occurred.
3. **Intermediate commit** (optional if within the same wave, mandatory between waves).

### Rule: re-read when context degrades

If during the session the agent notices signs of context degradation:

- Repeats instructions already present in files
- Asks things that L0/L1/L2 already explain
- Operates outside the protocol (ignores stop points, skips TDD cycle, does not use subagents)
- Does not remember the correct branch or current wave

The agent MUST immediately stop and execute a **context refresh**:

1. Re-read L0 (`CLAUDE.md`), L1 (`CONTEXT.md`), L2 (`stages/<stage>/CONTEXT.md`).
2. Re-read `_kickoff.md`.
3. If context is already ≥70% (detected via automated hook or heuristic estimate): execute an early handoff — update `_kickoff.md` and L1, commit, print KICKOFF block, and EXIT the session.
4. If context seems <70% but degraded: continue after re-reading.

### Rule: subagents mandatory in waves with 2+ tasks

In **stage 04**, when the wave has 2 or more tasks, the lead MUST use parallel subagents in isolated branches as per `subagent-protocol.md`. Running all tasks sequentially in a single session is an **anti-pattern** that amplifies context pressure and ignores the protocol's parallelism.

Cap per tier (Q17):
- experimental: 2 subagents
- tool: 3 subagents
- development: 5 subagents
- production: 5 subagents

### Automated hook: context-check.sh

The project includes a `PostToolUse` hook that automatically detects context ≥70%:

- **File:** `<project_root>/workspaces/<NNN-slug>/.claude/hooks/context-check.sh`
- **Registration (two points, both written by bootstrap):**
  1. `<project_root>/.claude/settings.local.json` → `hooks.PostToolUse` with command `bash "$CLAUDE_PROJECT_DIR/workspaces/<NNN-slug>/.claude/hooks/context-check.sh"` (project_root scope, primary registration).
  2. `<project_root>/workspaces/<NNN-slug>/.claude/settings.local.json` → `hooks.PostToolUse` with command `bash "$CLAUDE_PROJECT_DIR/workspaces/<NNN-slug>/.claude/hooks/context-check.sh"` (workspace scope, redundancy for cases where Claude Code reads settings from subdir).
- **Absolute path via `$CLAUDE_PROJECT_DIR`:** Claude Code exposes this env var to the hook process, always pointing to project_root. Required because session may run with cwd ≠ project_root (worktree `.icm-main/`, subdir). Relative path `bash workspaces/.../hook.sh` breaks with `bash: workspaces/.../hook.sh: No such file or directory`. Double quotes around the path = mandatory (Windows paths with spaces).
- **Logic:** reads transcript directly from `~/.claude/projects/`, computes `ctx_pct` (independent of statusline.sh), threshold 70%, cooldown 60s
- **Action:** stdout = mandatory handoff protocol. Agent sees the message and MUST stop everything, execute handoff, and EXIT.
- **Bootstrap:** installed by ICM stage 00 as governance infrastructure. Dependency `jq` in `_config/xp-conventions.md`. Template bytes normalized CRLF→LF on copy (kernel exec of shebang fails on CRLF — looks for interpreter `bash\r`).
- **Scope:** runs for the main session (lead) at project_root. Subagents in ephemeral worktrees (`Agent(isolation: "worktree")`) do **NOT** receive the hook — `<project_root>/.claude/settings.local.json` is not accessible to the worktree, and `workspaces/` does not exist on branch `wave-N-M/<task>`. Subagent auto-compact is accepted by design (target token budget §9 of `subagent-protocol.md` keeps ctx away from threshold).

### Additional hook — tier=production: block-dangerous-git.sh

Workspaces created with `tier=production` receive a 4th entry in `hooks.PreToolUse` (matcher `Bash`) registering `block-dangerous-git.sh`. Blocks `git push --force`, `reset --hard`, `clean -fd`, `branch -D`, `checkout/restore .` before execution. Hook .sh is copied by bootstrap (see `_PRODUCTION_HOOK_FILES`) and registered by `_merge_project_settings_local(..., tier="production")`. On other tiers, hook is not copied or registered.

The hook does NOT force a stop via code (exit 0 = continue). Enforcement is by protocol: agent instructed to obey the message. If the agent repeatedly ignores it, the LLM's compaction eventually degrades context beyond recovery — the hook keeps firing each tool call with a 60s cooldown.

### Template: mid-wave _kickoff.md update

After each completed task, append to `_kickoff.md`:

```yaml
prev_outputs:
  - path: "stages/04_implementation_waves/output/wave-N/task-slug.md"
    summary: "description of completed task (N tests, ruff green)"
```

And update `pending_for_this_stage` removing completed tasks.

### Anti-patterns

#### Continuing session after losing governance

If the agent lost L0/L1/L2 from context and did not execute refresh, subsequent operations likely violate the protocol (wrong branch, skip TDD, no stop points). Prevention is better than recovery — checkpoint after each completed task.

#### Running entire wave without subagents

In waves with 2+ tasks, each task MUST have its own subagent in an isolated branch. Lead orchestrates, does not execute. Single sequential session is allowed ONLY when the wave has 1 task (skip exception F2 of subagent-protocol).
